Coding agents harness LLMs. But who harnesses these coding agents?

A self-organizing harness that works alongside your existing hooks and keeps your agent focused, updated, reminded, refueled, and learning over time — automatically.

Currently works with Claude Code only. Support for other coding agents is on the way. Use at your own risk (and enjoyment). All comments and suggestions are welcome.

The Stack

A harness for the harness

Claude is an LLM. Claude Code is a harness that makes Claude useful as a coding agent. Harness Harness is a harness for Claude Code — it makes the coding agent itself better over time.

The foundation model

Claude Code already has a hook system — lifecycle events where you can inject context, trace tool calls, and react to changes. Harness Harness plugs into this hook system to add five capabilities that the agent doesn't have on its own:

Architecture

How it all fits together

Harness Harness creates .harness/ for its own state (routes, memory, sessions) and installs hh- prefixed hooks into .claude/hooks/. It works alongside your existing hooks — never replaces them unless you ask.

Works Alongside, Not Instead Of

.harness/ is your configuration. Route configs, custom intents, effectiveness scores. You edit these. You commit these. They define what context gets loaded for each type of task.

.claude/hooks/ is the runtime. All HH hooks are prefixed with hh-. Your existing hooks remain untouched and continue to fire alongside HH hooks. Claude Code runs all hooks for an event in parallel — the most restrictive decision wins.

Coexistence Guarantees

  • All HH hooks prefixed hh- — never collides with your hooks
  • init detects existing UserPromptSubmit hooks and warns before installing
  • Three modes: --merge (alongside), --replace (backup + replace), --trace-only (observe only)
  • Side-effect hooks (trace capture, session summary) are always safe — no stdout, no blocking

Hook Lifecycle

7 hooks that keep your agent
focused, updated, reminded, refueled

Claude Code fires hooks at specific lifecycle moments. Harness Harness listens at 7 of these events. Each hook is either a context injector (writes to stdout → Claude sees it) or side-effect only (writes to disk, never interferes with Claude).

Session Timeline

SessionStart context injector Updated + Refueled

hh-session-start.sh · matcher: startup|resume|compact · 5s

Creates session directory, injects git context (branch, uncommitted files, recent commits), and restores session state if resuming or recovering from compaction. Your agent starts every session knowing where it left off.

UserPromptSubmit context injector Focused + Reminded

hh-assembler.js + hh-assembler-fallback.sh · 10s + 5s

The Smart Context Assembler. Classifies your message's intent, loads the matching route config, fills budget slots with effectiveness-ranked rules, and outputs a <harness-context> block. If the assembler crashes, the fallback injects five critical rules so the agent is never completely unguided.

PostToolUse side-effect only Learning

hh-trace-capture.sh · matcher: (all tools) · 5s

Fires after every tool call. Captures one JSONL event: timestamp, tool name, input summary, output size, referenced rule IDs, files touched, and bash outcome. This trace data powers the effectiveness scoring and self-improvement loop.

PostToolUse additionalContext Updated

hh-state-nudge.sh · matcher: Edit|Write · 5s

Fires only on edits/writes. Returns a small nudge asking the agent to update its session state file. Keeps objectives, decisions, and blockers current across the session.

PreCompact side-effect only Updated

hh-pre-compact.sh · matcher: manual|auto · 5s

Saves a snapshot of session state to disk before context compression. Ensures trace data is flushed.

PostCompact context injector Refueled

hh-post-compact.sh · matcher: manual|auto · 10s

Re-injects identity, session state, and active decisions after compaction. This is the refueling moment — compaction becomes beneficial GC instead of information loss.

Stop side-effect only Learning

hh-session-summary.js · 30s

Reads trace JSONL, cross-references with the assembler manifest, scores every injected rule, computes slot utilization, and writes a structured summary. This is the raw data that feeds the self-improvement loop.

SessionEnd side-effect only

hh-session-end.sh · 10s

Safety net. If the Stop hook failed, this ensures the session is archived and trace data preserved.

Effectiveness Scoring

Every rule is measured.
Nothing runs on faith.

At session end, the summary hook reads traces and scores each injected rule:

0.0
ignored
Injected but never referenced. Wasted tokens.
0.5
implicit
Agent followed the rule without citing it.
1.0
referenced
Agent explicitly mentioned the rule ID.
2.0
prevented-mistake
Agent read anti-pattern file, then made an edit. The rule actively prevented a mistake.
Detection Logic
for (const ruleId of injectedRules) {
  // Was this rule ID mentioned in any tool response?
  if (traceEvents.some(e => e.referenced_context.includes(ruleId)))
    score = 1.0;  // referenced

  // Was an anti-pattern file read, then an edit followed within 3 events?
  for (let i = 0; i < traceEvents.length; i++) {
    if (traceEvents[i].tool === 'Read'
      && traceEvents[i].input_summary.includes('anti-pattern')
      && traceEvents[i].referenced_context.includes(ruleId)) {
      const next3 = traceEvents.slice(i+1, i+4);
      if (next3.some(e => e.tool === 'Edit' || e.tool === 'Write'))
        score = 2.0;  // prevented-mistake
    }
  }
}

Self-Improvement Loop

How the harness learns

This is the self-organizing mechanism. After accumulating session data, the system analyzes which rules help, which waste tokens, and proposes changes to its own configuration:

Per-Session automatic

Every session end: traces scored, utilization computed, summary written.

Per-Week automatic

Aggregate scores, generate proposals, auto-reorder by effectiveness.

Per-Month human review

Review proposals, approve/reject promotions and demotions.

Step 1

Observe

Trace every tool call. Record which rules are referenced.

Step 2

Analyze

Aggregate with exponential moving average. Recent sessions weighted 2x.

Step 3

Propose

Generate promotions, demotions, budget changes based on thresholds.

Step 4

Review

Human approves or rejects. Rejected items never re-proposed.

Step 5

Apply

Update route configs. Commit to git. Next session is better.

← Next session uses improved harness →

Safety Constraints

  • Never auto-remove rules — only humans can move items to Skip
  • prevented-mistake (2.0) protects safety nets — a rule with avg 0.0 but any prevented-mistake event is never proposed for demotion
  • Rejected proposals persist — recorded with reason, never re-proposed
  • Only reordering is auto-applied — no additions, removals, or section moves
  • All changes committed to git — full history of every evolution
  • Every mutating command supports --dry-run

--merge

Install alongside existing hooks. Both fire in parallel. User accepts potential double context.

Best for: projects with non-conflicting hooks

--replace

Back up existing UserPromptSubmit hooks to .harness/hooks-backup/, replace with assembler.

Best for: migrating from manual hooks

--trace-only

Only install tracing. No context injection. Keep your system but gain effectiveness scoring.

Best for: evaluating before committing

Getting Started

Install & Commands

# Install globally
npm install -g harness-harness

# Or run without installing
npx harness-harness init

# Initialize in your project
cd my-project
harness-harness init

# Use Claude Code normally — tracing starts automatically

# After a few sessions, check your dashboard
harness-harness health

# Run analysis
harness-harness analyze

# Apply approved proposals
harness-harness apply
